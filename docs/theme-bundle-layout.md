# Theme bundle layout

How an Aaron UI theme bundle is organized on disk. Pairs with [`docs/kaleidoscope-geometry-spec.md`](./kaleidoscope-geometry-spec.md) §7 (the `theme.json` schema) and [`docs/runtime-rendering-architecture.md`](./runtime-rendering-architecture.md) (how the runtime composites it).

## Directory tree

A canonical bundle lives at `themes/<scheme-slug>/`:

```
themes/<scheme-slug>/
├── theme.json          # schema-conformant bundle manifest (generated)
├── meta.json           # hand-authored provenance sidecar (regenerator input)
├── PROVENANCE.md       # human-readable origin + license record
├── cicns/              # chrome bitmaps (color icon resources)
│   ├── cicn-n14335-active-document-window.png
│   ├── cicn-n14336-inactive-document-window.png
│   └── ... (one PNG per cicn resource in the source scheme)
└── ppats/              # tile patterns (pixel pattern resources)
    ├── ppat-n129-titlebar-pinstripe.png
    └── ... (one PNG per ppat resource)
```

## Files

### `theme.json`

Generated by `scripts/build-theme-bundles.mjs` (which calls the extractor's `buildThemeJson` and validates via `validateTheme`). Always conformant to the [`docs/kaleidoscope-geometry-spec.md`](./kaleidoscope-geometry-spec.md) §7 schema. **Never hand-edit** — regenerate from `meta.json` + the extraction manifest.

Asset paths inside `theme.json` are bundle-relative — e.g. `chromeElements[*].asset = "cicns/cicn-n14335-active-document-window.png"`. The runtime resolves them against the bundle's served URL.

### `meta.json`

Hand-authored sidecar carrying everything the binary `.ksc` doesn't: bundle name, author, license, source URL. Fed to the extractor via `scheme-extract --meta meta.json` (or merged by `scripts/build-theme-bundles.mjs` on regeneration). Shape:

```json
{
  "name": "mass:werk 7 Le",
  "author": {
    "name": "Norbert Landsteiner",
    "email": "info@masswerk.at",
    "url": "https://www.masswerk.at",
    "year": 2001
  },
  "origin": {
    "kind": "kaleidoscope-port",
    "originalFormat": "ksc",
    "originalLicense": "Freeware. Distribute as long as with this readme file.",
    "originalReadme": "ReadMe-masswerk7Le",
    "sourceUrl": "https://www.masswerk.at/schemes.php"
  }
}
```

`originalLicense` should be the verbatim license string from the scheme's own readme — not a SPDX identifier or our paraphrase. Keep the source's words intact.

### `PROVENANCE.md`

Human-readable companion: original author, source URL, readme excerpt, our license interpretation, and why this scheme is in the corpus. Markdown so it renders on GitHub. See `themes/masswerk-7-le/PROVENANCE.md` as the canonical example.

The `PROVENANCE.md` and `meta.json` carry the same factual info in different forms — markdown for humans, JSON for the runtime. They must agree. If they ever drift, `PROVENANCE.md` is authoritative (it's the human-curated record) and `meta.json` should be corrected to match.

### `cicns/*.png` and `ppats/*.png`

Extracted by `tools/scheme-extractor/`. PNG filenames preserve the `cicn-n<id>-<slug>.png` and `ppat-n<id>-<slug>.png` convention so the resource ID is recoverable from the filename. (`n<id>` is the Kaleidoscope-style negative ID rendered as e.g. `n14335` for resource `-14335`.)

PNGs are 32-bit RGBA; `cicns` carry alpha-channel transparency, `ppats` are fully opaque tiles.

## URL conventions at runtime

When the runtime loads a bundle via `loadTheme("/themes/masswerk-7-le/")`:

1. Fetches `/themes/masswerk-7-le/theme.json`
2. For each `asset:` field in the JSON, resolves against the bundle URL — e.g. `cicns/cicn-n14335-active-document-window.png` becomes `/themes/masswerk-7-le/cicns/cicn-n14335-active-document-window.png`
3. PNGs stream as the runtime references them — no pre-fetch of every asset

This URL-relative scheme means a bundle works identically from any path: served by Vite at `http://localhost:5173/themes/...`, deployed to GitHub Pages at `https://owner.github.io/repo/themes/...`, or shipped inlined inside the npm package (Phase 4.5).

## Regenerating a bundle

```sh
# Regenerate one bundle (after updating its meta.json or re-running the extractor):
node scripts/build-theme-bundles.mjs masswerk-7-le

# Regenerate all canonical bundles:
node scripts/build-theme-bundles.mjs
```

The script reads `demo/assets/themes/<slug>/extraction-manifest.json` (the extractor's raw output, committed alongside the demo PNGs), copies the PNGs into the canonical `themes/<slug>/cicns/` and `ppats/` subdirs, runs `buildThemeJson` with the bundle's `meta.json` merged in, validates via `validateTheme`, and writes `theme.json`.

If validation fails the script aborts non-zero and does not write a partial `theme.json` — the bundle stays as it was.

## Porting a new scheme

The full porting flow is in [`CONTRIBUTING.md`](../CONTRIBUTING.md#adding-a-theme-porting-a-kaleidoscope-scheme). Short version:

1. Verify the scheme's readme grants redistribution rights.
2. Extract via `scheme-extract --input scheme.r --output demo/assets/themes/<slug>/`.
3. Create `themes/<slug>/` with `meta.json` + `PROVENANCE.md`.
4. Run `node scripts/build-theme-bundles.mjs <slug>` to materialize the canonical bundle.
5. Open a PR with a side-by-side screenshot of Aaron UI's render vs. the scheme's own preview thumbnail.

## What's *not* in a bundle

Per LEARNINGS 2026-05-16 "Themes don't bring sounds or desktop backgrounds":

- No `sounds/` — Kaleidoscope schemes in practice didn't carry them; Aaron UI doesn't fabricate them.
- No `desktop.png` — same reason.
- No `fonts/` — Mac OS supplied system fonts; the corpus assumes them present.

A future `extras/` sidecar concept may permit opt-in sounds/desktop/fonts for first-party preset bundles (see PRD §What ported themes carry), but it's not a runtime built-in for Phase 4.

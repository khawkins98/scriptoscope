The kDEF replay engine

Posted May 22, 2026 by Ken · 9 minute read · 5 comments

The Mac OS 8 / 9 window decorator was a 68k WDEF resource. Apple shipped about a dozen of them; Kaleidoscope shipped its own, the kDEF, which let theme authors override the system's window-drawing routines entirely. To render a Kaleidoscope scheme today, you have to re-implement the kDEF.

Re-implementation doesn't mean transpilation. The kDEF is 68k assembly that calls into the Mac Toolbox; we have no Toolbox in a browser. What we do have is a SPECIFICATION the kDEF was authored against: a part-code jump table, a wnd# rect-list walked side-by-side, a nine-slice of every cicn. The replay engine is a clean-room implementation of the spec, with the decompiled 2.3.1 binary used purely as the test oracle.

The most surprising thing about the implementation: it's small. The compositor (src/composeChrome.ts) is ~600 lines, the part-code classifier ~150 lines, and the rest is glue. The hardest part wasn't the math; it was figuring out which part codes the corpus actually used, which ones were authored against an older spec, and which were dead in the original schemes.

If the model needs per-theme special cases, it has the model wrong.

That's the working rule. When a render looked off, the answer was always "the model is wrong," never "this scheme is weird." Eighteen schemes shipped without a single per-theme branch in the compositor. The faithfulness ledger has the catalogue of intentional divergences from the binary, each with a reason. Read it if you're porting a scheme.

// The part-code classifier is the load-bearing piece.
const classification = classifyPart(partCode, border, rect);
switch (classification.kind) {
  case 'FIXED':   drawFixedSlice(buffer, rect, slice); break;
  case 'STRETCH': drawStretchedSlice(buffer, rect, slice, stretch); break;
  case 'TILE':    drawTiledSlice(buffer, rect, slice); break;
  // ...
}

The kDEF was clever in a way that's lost on modern graphics: it baked the WINDOW-CHROME SHAPE into the same byte stream as the pixel art. A scheme's title bar isn't just an image; it's an image plus a recipe for how to stretch it across any window size. That's why a 1998 scheme renders correctly on a 1600px-wide window today, even though the author never imagined that resolution.

I'll write more about the part-code jump table next month. For now: if you've never read decompiled 68k, the surprise is that it's MORE readable than 2026 React.

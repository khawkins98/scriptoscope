Scheme Diagnostics

Per-scheme view of how Aaron UI decodes + renders a Kaleidoscope scheme. Shows the raw chrome cicns with parts-rect + recipe-position overlays, the classifier verdict, the derived slice geometry, and a live render with optional segment-outline highlighting. Use this to diagnose extraction-vs-rendering gaps: if the recipe data looks correct here but the live render is wrong, the bug is in the renderer. If the data itself is missing or wrong, the bug is in the extractor.

Reference

Period rendering of the scheme — what the author actually saw on the classic Mac. mass:werk schemes have curated thumbnails on file; exotic schemes link out to the Kaleidoscope Scheme Archive where you can browse the original previews.

Classifier verdict

Chrome cicns

Each chrome state cicn at 4× scale. Toggle overlays to see how the renderer interprets the bitmap. Hover the ⓘ icons for what each overlay shows and why it matters.

Recipes (wnd#)

For each side: list of {at, part} entries. at is the cicn-pixel position along the side axis. part matching a slug in the parts table = named. part not in the table = fill (tiles cicn pixels in the segment). Recipe entries are slice-boundary markers, not per-widget render commands — the renderer slices the cicn at the fill-zone boundaries and pins left/right slices to corners at native size.

Parts (named cicn rects)

Slugs referenced by recipe entries. Each rect [left, top, right, bottom] in cicn pixels.

Per-segment inspector

Only meaningful for rich-recipe schemes (1990, Acid, evolution). For each edge, shows every segment the composer emits: its cicn slice preview, span, render behavior, and (the key column) how many times it would tile at the live-render window width. Tile-count > 1 on a slice that looks like a static graphic is the signal that we'd benefit from per-segment static-vs-tile metadata. See the column legend below.

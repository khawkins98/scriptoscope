Eighteen Kaleidoscope schemes, running in a browser tab twenty-five years too late.

Kaleidoscope was a control panel some of us couldn't leave alone. Greg Landweber and Arlo Rose wrote it; the rest of us kept a Scheme Folder we'd refresh every weekend because Hayato Mori or Erik Ekengren or somebody on Info-Mac had just posted a new one. Then OS X arrived and the whole format went quiet.

Scriptoscope is a compatibility layer that reads those same scheme files — the actual .sit and .rsrc bundles people made in the nineties — and paints them onto your webpage. Same color icons, same window frames, same fill patterns. Your HTML stays live underneath; only the skin is canvas.

A Kaleidoscope scheme is a Mac resource fork: a little database of typed records authored against a system that stopped shipping in 2001. So Scriptoscope reads one the way the old Finder did. Drop a scheme on the page, and it unpacks the archive, walks the resource map, pulls each record into memory, and hands the lot to a <canvas> renderer that speaks the same scheme format the original control panel did. (That renderer is a spec-compatible reimplementation of the 2.3.1 engine; the divergences are logged in the ledger if you care.) All of it runs in the tab. Nothing's uploaded, and a refresh throws the bytes away.

To skin your own page, tag an element for what it is — window, button, slider — add one script tag, and it wears the current scheme. No framework, no build step. The children stay real HTML; only the chrome is painted.

Why? No good reason. Kaleidoscope lost to OS X a quarter-century ago and the world was right to move on. I rebuilt the rendering layer anyway, so eighteen schemes by people who mostly stopped making them in 2002 could run in a browser tab. (The full corpus — 4,400-plus themes from 900-plus authors — is still out there if you want to dig.) This is a deeply silly use of anyone's time, including yours. If you're unhinged enough to ship it on a real website, I genuinely want to hear about it.

P.S. The repo is named aaron-ui because Aaron was Landweber's earlier extension — the one that faked the Mac OS 8 Platinum look on System 7 before Kaleidoscope existed. Three people will catch that. Hello, three people.

One engine, eighteen looks

Bring your own scheme

A Kaleidoscope scheme is a Mac resource fork: a tiny relational database of typed records (cicn, ppat, wnd#, cinf, Colr, clut, ics4, ics8) authored against a Mac OS that hasn't shipped since 2001. To get one onto the web you have to read it the way the Finder read it.

So that's what this does. Drop a .sit on the page — say, the original beosr503.sit Jon Alexander uploaded in 2002 — and the runtime hands the bytes to a WebAssembly StuffIt decoder, lifts the resource fork out of the unpacked MacBinary wrapper, walks the resource map, and pulls each typed record into memory. The 8-bit cicn color icons get their CLUTs applied and Mac 1.8 → sRGB 2.2 gamma-corrected; the wnd# rect list becomes the four-sided recipe the compositor walks part-code by part-code; the ppat patterns become tile sources for the desktop and the dialog grays. All of it in your browser — no upload, no server, no telemetry, no account. Refresh the tab and the bytes are gone.

Two things this enables that nothing else does. First: the eighteen bundled schemes ship as their original archives — scheme.sit straight from kaleidoscope.net's 1999 mirror, or scheme.rsrc recovered from a Wayback crawl when the StuffIt is no longer reachable — with a meta.json that records who made it and where it came from. Nothing's been re-encoded or “cleaned up.” If a 1998 byte was wrong, it's wrong here.

Second: the demo's drop zone is the same code path as the bundled corpus. Drag any of the 3,000-plus schemes from the Kaleidoscope Scheme Archive onto the page and watch it render — the runtime doesn't know or care that you didn't ship it.

The decoder lives in tools/theme-loader/ and has no dependency on the renderer, the DOM, Node, or anything else app-specific. If you want to read resource forks in a browser for some other reason — a Mac archaeology project, a Hypercard stack viewer, a Mac OS save-game inspector — it's MIT licensed and it'll work.
